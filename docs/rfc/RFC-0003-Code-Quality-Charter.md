# RFC-0003: Code Quality Charter

## 状态

Draft

## 总则

本章程用于定义 AI 生成代码与人工协作代码的质量基线。

核心原则是：

- 企业项目优先
- 兼容性优先
- 可维护性优先
- 最小修改优先
- 文档与代码同步

## 编码原则

- 优先复用已有实现
- 不重复造轮子
- 不为了“更优雅”而增加不必要抽象
- 不随意重构历史代码
- 不随意新增依赖
- 不在未确认上下文时改动公共行为

## 注释规范

### 方法注释

所有新增的公共方法、复杂业务方法和接口方法必须提供准确、简洁的方法注释。

方法注释应说明：

- 方法的业务功能
- 关键参数含义
- 返回值含义
- 异常或副作用

### 行级注释

在以下场景必须补充简洁行级注释：

- 复杂条件判断
- 兼容历史逻辑
- 非直观数据转换
- 金额、库存、优惠券、权限等关键业务
- 事务、锁、幂等、重试
- 特殊绕过逻辑

行级注释只描述“为什么这样做”或“这段代码完成什么功能”，不要写冗长重复的解释。

### 注释准确性

修改代码时，必须同步检查并更新相关注释，禁止保留过时或不准确的注释。

## 日志规范

- 关键业务流程必须记录日志
- 错误日志必须包含足够上下文
- 日志内容要可定位问题
- 不要输出敏感信息
- 不要使用无意义的调试输出代替日志

## PSR-4 与 PSR-12

### PSR-4

在项目已经使用 Composer 和命名空间的情况下，新建类必须遵循现有 PSR-4 自动加载映射。

### PSR-12

在项目运行版本允许的前提下，新代码应尽量遵循 PSR-12 代码风格。

### 兼容性例外

对于历史项目，必须以项目现有风格与兼容性为先，不得为了统一规范而批量破坏原有结构。

## 兼容性优先规则

- 先确认 PHP 版本，再决定语法
- 先确认 ThinkPHP 版本，再决定框架 API
- 不允许将新版本语法误用到旧版本项目
- 不允许将框架升级假设当作事实

## 企业开发原则

- Controller 保持轻量
- 业务逻辑放在 Service 或等价层
- 数据访问职责要清晰
- SQL 变更必须考虑索引、回滚和影响面
- Redis 使用必须考虑前缀、TTL、热点与一致性
- 接口必须考虑幂等、鉴权、参数校验和错误码
- 所有关键能力都应可测试、可审查、可回溯
